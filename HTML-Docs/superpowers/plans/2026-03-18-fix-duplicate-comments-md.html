Fix: Duplicate Comments MD Generation Implementation Plan


> For agentic workers: REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (- [ ]) syntax for tracking.


Goal: Remove duplicate -Comments.md files caused by fetch_comments being called twice per article — once inside fetch_article_content and again by the outer caller.
Architecture: Both HnSource.fetch_article_content and LbSource.fetch_article_content call self.fetch_comments() internally. But unified_source_processor._process_single_article_new_system (the caller) also calls source.fetch_comments() after fetch_article_content returns. This produces two comments files per article with different names (first uses the web page title; second uses article.title from the source). Fix: remove the embedded fetch_comments calls from both source fetch_article_content methods. The outer caller already handles comments correctly.
Tech Stack: Python, pytest, unittest.mock

Root Cause Summary


unified_source_processor.py:_process_single_article_new_system (lines 718–752): 
success, article_path = source.fetch_article_content(article, base_dir, progress_callback)
# ↑ HnSource/LbSource fetch_article_content calls self.fetch_comments() inside ← FIRST WRITE

if supports_comments and hasattr(source, "fetch_comments"):
    source.fetch_comments(article.comment_url, article.title, article_path)  # ← SECOND WRITE

Two different titles → two different filenames in the article folder.

Context for Subagents


What was changed and why (titled markdown filenames feature — v1.0.45)


The previous session (feat/titled-markdown-filenames, PR #8) changed article.md / comments.md to use article titles as filenames. After that change, this duplicate bug became visible: two comments files with slightly different names appear in each HN/LB article folder.


Why the bug exists


HnSource.fetch_article_content and LbSource.fetch_article_content both:
  1. Call fetcher.fetch_article_content() (the ArticleFetcher method) to download the article
  2. Then call self.fetch_comments() inside the same method — this is the first comments write
  3. Return (True, folder_path)

The caller, unified_source_processor._process_single_article_new_system, then checks hasattr(source, "fetch_comments") and calls it again — this is the second comments write.


The two calls use different titles:

  • First (internal): title or article.title where title = web page title (e.g. "'The Secret Agent' Exploring a Vibrant, yet Violent Brazil 2025")
  • Second (external): article.title (HN scraper title, e.g. "'The Secret Agent' Exploring a Vibrant, Yet Violent Brazil")

Result: two different -Comments.md files in the same folder.


Why removing the internal call is safe


unified_source_processor._process_single_article_new_system already handles comments correctly:
  • It uses article.title (consistent with the article folder name)
  • It uses article_path (the folder path returned by fetch_article_content)
  • It wraps in try/except with proper logging

Removing the internal call does not lose any functionality.
