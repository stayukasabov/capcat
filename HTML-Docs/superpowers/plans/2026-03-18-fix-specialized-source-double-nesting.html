Fix: Specialized Source Double-Nesting Implementation Plan


> For agentic workers: REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (- [ ]) syntax for tracking.


Goal: Remove the double-nested folder created when processing YouTube/Medium/Substack articles — content must land in base_folder/VideoTitle/ not base_folder/HN_title/VideoTitle/.
Architecture: UnifiedArticleProcessor._process_with_specialized_source creates an intermediate folder using the HN post title, then passes that as output_dir to the specialized source. The specialized source (YouTube, Medium, Substack) ALSO creates a subfolder inside output_dir using the actual content title. Result: two levels of nesting. Fix: remove intermediate folder creation for non-update_mode; pass base_folder directly so the specialized source's subfolder is the only one created.
Tech Stack: Python, pytest, unittest.mock

Root Cause Summary


unified_article_processor.py:_process_with_specialized_source (lines 118–154): 
# Lines 118–121 — PROBLEM: creates intermediate folder with HN title
safe_title = sanitize_filename(title or source_id)
article_folder = os.path.join(base_folder, safe_title)
os.makedirs(article_folder, exist_ok=True)

# Line 140–142 — passes intermediate folder to source
success, article_title = source_instance.fetch_article_content(
    article, article_folder, progress_callback=None   # ← intermediate, not base_folder
)

# Line 149 — returns intermediate folder, not what source created
return True, article_title, article_folder           # ← article_folder = intermediate (wrong)

YouTube (and Medium/Substack) do:

article_folder = os.path.join(output_dir, safe_title)   # creates subfolder inside output_dir
os.makedirs(article_folder, exist_ok=True)
return True, article_folder                              # returns that subfolder

Combined effect: base_folder/HN_title/YouTube_title/ instead of base_folder/YouTube_title/.

def _make_processor() -> UnifiedArticleProcessor:
    return UnifiedArticleProcessor()


def _make_mock_specialized_source(tmp_path, content_folder_name="YouTube-Video-Title"):
    """Return a mock specialized source that creates a real subfolder in output_dir."""
    def _fake_fetch(article, output_dir, progress_callback=None):
        content_folder = os.path.join(output_dir, content_folder_name)
        os.makedirs(content_folder, exist_ok=True)
        return True, content_folder

    source_instance = MagicMock()
    source_instance.fetch_article_content.side_effect = _fake_fetch
    return source_instance

class TestNoIntermediateFolder:
    """_process_with_specialized_source must not create base_folder/<hn_title>/."""

    def test_no_intermediate_folder_created(self, tmp_path):
        """The single direct child must be named after the content, not the HN post title."""
        content_name = "YouTube-Video-Title"
        hn_title = "Some HN Post Title"
        processor = _make_processor()
        source_instance = _make_mock_specialized_source(tmp_path, content_name)

        with patch.object(
            processor.specialized_manager,
            "can_handle_url",
            return_value=True,
        ):
            with patch.object(
                processor.specialized_manager,
                "get_source_for_url",
                return_value=(source_instance, "youtube"),
            ):
                success, title, folder = processor.process_article(
                    url="https://www.youtube.com/watch?v=abc123",
                    title=hn_title,
                    index=0,
                    base_folder=str(tmp_path),
                )

        assert success is True

        # The single direct child must be the content folder, not an HN-title intermediate
        children = [p for p in tmp_path.iterdir() if p.is_dir()]
        assert len(children) == 1, (
            f"Expected exactly 1 subfolder in base_folder, found: "
            f"{[p.name for p in children]}"

            f"Expected single child to be '{content_name}' (content folder), "

        title: str,
        base_folder: str,
        update_mode: bool

        """

        Args:
            url: Article URL
            title: Article title
            base_folder: Output directory
            update_mode: Whether to re-check URL and update timestamp

        Returns:
            Tuple of (success, article_title, article_folder_path)

        source_result = self.specialized_manager.get_source_for_url(url)

        if not source_result:

        # Fetch content with specialized handler
        success, article_title = source_instance.fetch_article_content(

Context for Subagents


What the double-nesting bug looks like


When a Hacker News article links to a YouTube video, the current flow is:


  1. unified_source_processor calls UnifiedArticleProcessor.process_article(url="youtube.com/...", title="Some HN Title", base_folder="/output/HN_date/")
  2. process_article calls _process_with_specialized_source
  3. _process_with_specialized_source creates /output/HN_date/Some-HN-Title/ (intermediate)
  4. Passes that intermediate folder to YouTubeSource.fetch_article_content
  5. YouTube creates /output/HN_date/Some-HN-Title/Actual-YouTube-Video-Title/

Result: content is two levels deep. The intermediate folder is empty (or nearly empty).


What the fix achieves


After the fix:

  1. _process_with_specialized_source does NOT create the intermediate folder (in normal mode)
  2. Passes base_folder directly to the specialized source
  3. YouTube creates /output/HN_date/Actual-YouTube-Video-Title/ — correct

The intermediate folder only appears in update_mode=True for backward compatibility with existing articles.


Why update_mode keeps the intermediate folder


Update mode runs on articles that ALREADY EXIST on disk. Those articles were created with the old (buggy) structure. The intermediate folder IS there. _check_url_validity and _update_timestamp operate on that folder. Changing update_mode behavior would break existing article updates.


All three specialized sources follow the same pattern


youtube/source.py, medium/source.py, and substack/source.py all:
  • Accept output_dir as the second argument
  • Create a subfolder os.path.join(output_dir, safe_title) inside it
  • Return (True, subfolder_path)

The fix to _process_with_specialized_source fixes all three with one change.
