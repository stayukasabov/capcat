# Capcat

# PART TWO

## 01. Summary

The first version of Capcat worked. It fetched articles, organised them, generated clean HTML, and respected robots.txt. But it had two walls in front of it that I had not addressed. A new user could not install it without cloning the repository. An existing user who wanted to configure it had nowhere to put that configuration without editing the application itself.

I ran user testing with four people: two developers, one ML engineer, and one technically capable designer. They hit both walls in the first session. That testing defined the problem precisely enough to solve it. The solution was one architectural decision that removed both walls at the same time: separate the tool from the archive. The user owns the vault. Capcat is the tool that fills it.

**Role:** Product Designer and Developer
**Timeline:** Early 2026
**Team:** Solo project
**Tools:** Claude Code, Gemini, Figma, Drawio, Affinity Designer
**Process:** User testing, problem definition, vault architecture, iterative UX design across install, configuration, and interactive surface changes

---

## 02. The New Problem

Capcat worked if you already had it. Getting it required cloning the repository, navigating to the Application folder, and running `./capcat` from that exact path. One wrong directory and nothing worked. A developer who found capcat.org and wanted to try it had no clean path forward. The tool was effectively invisible behind its own setup process.

That was the external wall. There was also an internal ceiling. Users who had Capcat running and wanted to tune it — different article counts per source, per-source media settings, their own configuration — had nowhere to put that without editing the application's own files. News output, single article output, and the application itself lived in the same parent folder. The tool and the archive it created were peers. There was no separation between them.

Both walls existed at the same time. Removing one without the other would have solved half the problem.

## 03. Discovery and User Insights

I ran testing with four people before writing a line of new code. Two developers who work in the terminal daily. One ML engineer who uses arxiv.org extensively and archives research papers as part of a working process. One technically capable designer who uses Obsidian as their primary knowledge base.

The failures were fast and consistent. Every tester hit the installation wall in the first session. No one made it to configuration friction without help getting the tool running first.

Once past setup, the next wall was configuration rigidity. A developer wanted to set Hacker News to fetch fifty articles and IEEE to fetch five. There was no way to do that without editing the source config files inside the application directory. The ML engineer wanted PDF downloads from arxiv as a first-class feature, not a workaround. The Obsidian user wanted the articles Capcat fetched to appear as connected nodes in their knowledge graph — with tags, source metadata, and backlinks between articles and their comment threads.

Three friction walls. Four users. One architectural gap underneath all of them: the application and the user's content lived in the same place, and neither could be changed without touching the other.

## 04. The Vault

The solution was a single architectural decision. Separate the tool from the archive. The application installs globally via pipx into an isolated environment. The user works from their own directory — the vault — which holds everything that belongs to them: their sources, their configuration, their fetched content.

~/my-vault/

capcat.yml ← source list and per-source counts

Config/

Global-settings.yaml ← network, processing, logging, PDF settings

sources/

active/

hn/config.yaml ← per-source overrides

lb/config.yaml

News/ ← batch fetch output

Capcats/ ← single article output


`capcat init` scaffolds this structure on first run. It writes `capcat.yml` with the default source list and `Config/Global-settings.yaml` with the full settings template. Every key is commented inline. The user edits what they want, leaves what they do not.

The application can be upgraded, reinstalled, or uninstalled entirely. The vault survives unchanged. No sync layer. No cloud dependency. No service that can shut down and take the archive with it.

The same separation resolves the one-folder problem. Before, Application, News, and Capcats were peers in the same parent directory. After, the tool is global and the vault is wherever the user decides it should be — an existing Obsidian vault, a Synology drive, a dedicated folder on the desktop.

> **H3, User Control and Freedom:** Users often choose system functions by mistake and need a clearly marked emergency exit to leave the unwanted state without having to go through an extended dialogue.

The vault is unconditional user ownership. The tool is a visitor. That relationship is the design.

## 05. Configuration as UX

Every customization decision in Capcat resolves through four levels. The user can go as deep as they need. They never have to go deeper than they want.

The highest priority is the CLI flag. `--pdfs`, `--count 20`, `--media` override everything for the duration of one run. The second level is the TUI prompt — a select menu that sets policy for an interactive session. The third level is the per-source config file in `Config/sources/active/hn/config.yaml`. The fourth level is `Config/Global-settings.yaml`, the vault-wide defaults.

A user who never opens a config file gets sensible defaults for everything. A user who wants Hacker News to fetch fifty articles and IEEE to fetch five sets that once in two config files and never thinks about it again. A developer running a cron job overrides everything with CLI flags. The same four levels serve all three without compromise.

`capcat settings --force` regenerates the `Global-settings.yaml` template at any time. Every key is documented in the file. The user does not need to read external documentation to understand what a setting does.

> **H7, Flexibility and Efficiency of Use:** Accelerators, unseen by the novice user, may often speed up the interaction for the expert user such that the system can cater to both inexperienced and experienced users.

The complexity is the feature. The hierarchy exists so the simple case stays simple and the complex case stays possible.

## 06. New Interactive Moments

Three new surfaces appeared between v1.0 and v1.9. Each one follows the same constraint as Part One: technically capable users, no time, no tolerance for surprises.

**The PDF TUI Prompt**

When running interactively with sources that have PDFs, the TUI shows a speed warning before the decision point. The user sees three options and makes one choice. That choice sets the PDF policy for the entire run.

⚠ PDF downloads can significantly slow article fetching.

Recommended: use --pdfs flag for dedicated PDF runs.


Download PDFs for this session?

> Yes

No

Source defaults


"Source defaults" hands the decision back to the per-source `media:` block. The ML engineer who has configured arxiv to download PDFs and Nature to skip them gets exactly that without re-specifying it at the prompt.

> **H6, Recognition Rather Than Recall:** Minimize the user's memory load by making objects, actions, and options visible. The user should not have to remember information from one part of the dialogue to another.

Three options are on screen. The user does not need to remember the `--pdfs` flag. The async download runs in the background after fetch completes, logging a single line when state changes — started, completed, failed.

**Source Update Ownership**

Sources ship with Capcat. When a new version changes a source config, the application needs to update that config in the vault. But a user who has customised their `hn/config.yaml` should never have their changes silently overwritten.

Every source in the vault manifest carries an `ownership` field: `app` or `user`. App-owned sources update silently. User-owned sources surface an interactive prompt:

Source 'hn' config has been updated in the new version.

Your vault config differs from the new app default.


> Keep my version

Accept app update

Show diff


The user who never touched the config never sees this prompt. The user who customised it sees it exactly once per update, with three clear paths forward.

> **H9, Help Users Recognize, Diagnose, and Recover from Errors:** Error messages should be expressed in plain language, precisely indicate the problem, and constructively suggest a solution.

Nothing is decided without consent. Nothing is lost without knowledge.

**Frontmatter and Backlinking**

Every article Capcat writes includes YAML frontmatter:
yaml

title: The article title

source: hn

url: https://example.com/article

date: 2026-04-18

tags: [tech, programming]

comments: [[hn-comments-article-title]]


The comments file links back:
yaml

title: HN Discussion — The article title

article: [[article-title]]

source: hn


Obsidian reads both files natively. The knowledge graph connects the article to its comment thread. Comment author names are replaced with Anonymous at write time. Profile links are preserved for reference. Privacy is built into the output, not offered as a setting.

> **H2, Match Between System and the World:** The system should speak the users' language, with words, phrases, and concepts familiar to the user, rather than system-oriented terms.

Obsidian users already think in linked notes and graph views. Capcat writes the frontmatter and wikilinks that Obsidian already understands. The tool did not invent a linking system. It learned the language the user already spoke.

## 07. The Integrated Solution

From empty machine to running vault in four commands.

pipx install capcat

cd ~/my-vault

capcat init

capcat catch


`pipx install capcat` installs the tool globally into an isolated environment. No cloning. No path management. No Python version conflicts. A developer who has installed any modern Python CLI tool knows exactly what this does.

`cd ~/my-vault` is the user choosing where their vault lives. It can be their existing Obsidian vault. It can be a Synology drive. It can be any directory on any machine. The tool goes wherever the content needs to be.

`capcat init` scaffolds everything. `capcat.yml` with the default source list. `Config/Global-settings.yaml` with the full commented settings template. The directory structure for News and Capcats output. One command. The vault is ready.

`capcat catch` launches the TUI. The six-option main menu appears. The user picks a bundle, decides on HTML, reviews, confirms, and executes. The first batch of articles arrives in `News/news_18-04-2026/`.

From that point, every customisation path is open. Edit `Config/sources/active/hn/config.yaml` to tune per-source counts. Add `media: download_pdfs: true` to the arxiv source config for automatic PDF downloads. Pass `--pdfs` at runtime for a dedicated PDF run. Write a cron job: `cd ~/my-vault && capcat bundle tech --count 20 --html`. None of these require touching the application files. The vault handles everything the user owns.

## 08. Validation and Impact

**H4 — Consistency and Standards**

> **H4, Consistency and Standards:** Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform conventions.

`pipx install capcat` follows the installation pattern of every modern Python CLI tool. A developer who has installed `black`, `ruff`, or `httpie` knows exactly what to do. The learning curve is zero. Meeting the standard was the design decision.

**H3 — User Control and Freedom**

> **H3, User Control and Freedom:** Users often choose system functions by mistake and need a clearly marked emergency exit to leave the unwanted state without having to go through an extended dialogue.

The vault is unconditional. The user owns the directory. Capcat can be upgraded, replaced, or uninstalled. The archive survives anything because it is a filesystem directory, not a database inside the application.

**H7 — Flexibility and Efficiency of Use**

> **H7, Flexibility and Efficiency of Use:** Accelerators, unseen by the novice user, may often speed up the interaction for the expert user such that the system can cater to both inexperienced and experienced users.

Four config levels. The novice finds the defaults in `Global-settings.yaml` and changes one number. The power user maps every source to a specific count and media policy. The developer ignores config entirely and passes flags. The hierarchy is invisible until needed, present when required.

**H9 — Help Users Recognize, Diagnose, and Recover**

> **H9, Help Users Recognize, Diagnose, and Recover from Errors:** Error messages should be expressed in plain language, precisely indicate the problem, and constructively suggest a solution.

Source update conflicts surface a prompt with three choices: keep, accept, or diff. No update happens without consent. The user who never customised a source config never sees the prompt. The user who did sees it exactly once.

**H6 — Recognition Rather Than Recall**

> **H6, Recognition Rather Than Recall:** Minimize the user's memory load by making objects, actions, and options visible. The user should not have to remember information from one part of the dialogue to another.

The PDF prompt shows Yes, No, and Source defaults on screen. The user recognises their choice. The async drain logs one line per state change. The user can see when downloads start and when they finish without watching a progress bar.

**H2 — Match Between System and the World**

> **H2, Match Between System and the World:** The system should speak the users' language, with words, phrases, and concepts familiar to the user, rather than system-oriented terms.

Frontmatter and wikilinks are Obsidian's native language. Capcat writes in that language. The article-to-comment connection in the knowledge graph is not a Capcat feature — it is Obsidian doing what it already does, with content Capcat structured correctly for it.
