Per-Source Article Count & TUI Audit Implementation Plan


> For agentic workers: REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (- [ ]) syntax for tracking.


Goal: Give every source its own article_count field in its YAML config, wire it into the fetch pipeline, expose it in the add-source and list-sources TUI flows, remove the dead "Generate Custom Source Config" menu entry, and add test coverage for four TUI flows that have been untested since the pipx migration.
Architecture: article_count: 30 is added to all source YAMLs (builtin and custom). SourceConfig gets a first-class article_count: int = 30 field. process_source_articles accepts Optional[int] — when None it resolves from the source config. The CLI --count flag default changes from "30" to None to enable per-source resolution. Bundle default_count is removed everywhere — per-source count is the single source of truth.
Tech Stack: Python, PyYAML, questionary, pytest

def _resolve_count(
    cli_count: Optional[int],
    source_config: "SourceConfig",
) -> int:
    """Resolve article count: CLI flag > source YAML > default 30.

    Args:
        cli_count: Value from --count flag, or None if not provided.
        source_config: The source's SourceConfig (has article_count field).
