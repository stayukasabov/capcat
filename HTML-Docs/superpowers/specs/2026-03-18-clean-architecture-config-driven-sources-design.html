Clean Architecture: Config-Driven Sources & Comment Processor Design


Date: 2026-03-18 Status: Approved

Goal


Remove all source-specific knowledge from the comment processor, delete the legacy adapter chain, and confirm that YAML auto-registration (already implemented) is the sole path for config-driven sources — so that adding a new news source requires only a YAML file and zero Python.



Motivation


After several migration cycles, the codebase has accumulated three layers of dead weight:


  1. StreamlinedCommentProcessor contains 5 source-specific wrapper methods — each one is a selector config embedded in Python that belongs in each source's own code.
  2. A three-layer legacy adapter chain (ConfigurableNewsSource → ModularNewsSource → NewsSourceAdapter) bridges YAML configs into the old processing path. It co-exists with the working new-system path despite being entirely superseded.
  3. _process_article_generic in USP routes BBC/Guardian/IEEE through this legacy chain _unless_ _is_source_in_new_system() returns True — which it already does, since ConfigDrivenSource and YAML auto-registration are already implemented. The legacy path is reachable only if the registry scan fails.


Scope


In scope


  1. StreamlinedCommentProcessor cleanup — remove 5 source-specific wrapper methods; each source owns its selector config.
  2. Dead code deletion — broken/unreachable symbols confirmed in prior audit.
  3. Verify ConfigDrivenSource routing — confirm bbc/guardian/ieee route through _process_with_new_system correctly (the implementation exists; this is verification + hardening).
  4. Legacy adapter chain deletion — NewsSourceAdapter, ConfigurableNewsSource, ModularNewsSource, related USP methods; safe only after step 3 is verified.
  5. Archive capcat/htmlgen/ — move to Archive/htmlgen/; confirmed zero imports of capcat.htmlgen in source tree.

Out of scope


  • TUI / CLI source management commands (capcat add-source, capcat list-sources)
  • HTML generation refactor (documented in docs/architecture/roadmap.md)
  • New source additions


Architecture


ConfigDrivenSource — already exists


capcat/core/source_system/config_driven_source.py implements ConfigDrivenSource(BaseSource) with discover_articles, fetch_article_content, and _validate_custom_config. The source registry (source_registry.py) already calls _discover_config_driven_sources() at init, scanning config_driven/configs/*.yaml and registering each as a ConfigDrivenSource.

The architecture task is verification and legacy deletion, not implementation:


source_registry init (already implemented) ├── scan custom/*/source.py → register HN, LB, LessWrong └── _discover_config_driven_sources() → scan config_driven/configs/*.yaml → ConfigDrivenSource(config) registered under config.name

Pre-deletion check required: verify _is_source_in_new_system() returns True for every source name reachable through process_source_articles. Any source in is_source_configured() that is absent from the new registry would fall through to the legacy path after deletion and fail silently. An explicit startup assertion gates the deletion.


StreamlinedCommentProcessor cleanup


Remove all 5 source-specific wrapper methods. The processor exposes only:

  • process_comments_flattened(soup, comment_selector, user_selector, comment_text_selector)
  • generate_inline_comments_markdown(comments, title, url, article_folder_path=None)
  • generate_inline_comments_html(comments, title, url)

hn/source.py currently calls both a markdown and an HTML wrapper. Both are replaced: 
HN_SELECTORS = {"comment_selector": ".comment-tree .athing",
                "user_selector": ".hnuser", "comment_text_selector": ".comment"}

# Markdown branch (before)
content = processor.process_hacker_news_comments_optimized(soup, title, url, folder)
# Markdown branch (after) — note: article_folder_path (4th arg) is passed here only
content = processor.generate_inline_comments_markdown(
    processor.process_comments_flattened(soup, **HN_SELECTORS), title, url, folder
)

# HTML branch (before)
content = processor.process_hacker_news_comments_html_optimized(soup, title, url)
# HTML branch (after) — no article_folder_path; HTML output does not embed wikilinks
content = processor.generate_inline_comments_html(
    processor.process_comments_flattened(soup, **HN_SELECTORS), title, url
)

lb/source.py follows the same pattern with Lobsters selectors: 
LB_SELECTORS = {"comment_selector": ".comment",
                "user_selector": ".user", "comment_text_selector": ".comment_text"}

LessWrong has no source.py in the codebase. process_lesswrong_comments_optimized has zero callers in the source tree (confirmed by grep). Its deletion is safe with no corresponding file update required.
Existing test in tests/test_lobsters_comment_rate_limiting.py mocks process_lobsters_comments_optimized directly on the processor object — that mock target no longer exists after deletion. The test must be updated to mock process_comments_flattened instead. Important: the return value type changes. The old wrapper returned a str (fully formatted markdown). process_comments_flattened returns list[dict]. The mock must return a list[dict] and the test assertion must account for the two-call chain (process_comments_flattened → generate_inline_comments_markdown) producing the final str.

Data Flow After Migration


All sources route through _process_with_new_system:


capcat fetch bbc
  → USP.process_source_articles("bbc")
  → _is_source_in_new_system("bbc") → True
  → _process_with_new_system("bbc")
  → ConfigDrivenSource.discover_articles(n)
  → ConfigDrivenSource.fetch_article_content(...)
  → fetch_comments → False (no comments for BBC)

The _process_article_generic branch in USP is unreachable for any registered source once the invariant check passes.



Testing Strategy


  • Regression: All existing HN and LB comment tests pass after wrapper removal.
  • Regression: test_lobsters_comment_rate_limiting.py passes after mock target update.
  • Invariant check: Test that _is_source_in_new_system returns True for bbc, guardian, ieee, hn, lb before legacy deletion.
  • ConfigDrivenSource routing: Integration test with bbc config and mocked HTTP confirms articles are discovered and content fetched through the new path.
  • Dead code: After deletion, grep -r "_process_article_generic\|NewsSourceAdapter\|ConfigurableNewsSource" returns zero results in capcat/.


Migration Sequence


Steps are independently committable. Step 3 is gated on step 2 passing.


  1. StreamlinedCommentProcessor cleanup — remove 5 wrappers, update hn/source.py, lb/source.py, fix rate-limiting test. All comment tests green.
  2. Verify ConfigDrivenSource routing — add invariant assertion that every is_source_configured() source is in the new registry. Confirm bbc/guardian/ieee return True from _is_source_in_new_system. Write routing integration test.
  3. Delete legacy adapter chain — news_source_adapter.py, source_factory.py, sources/base/factory.py, sources/base/config_schema.py, dead USP methods. Full test suite green.
  4. Archive capcat/htmlgen/ — confirmed zero imports; move to Archive/htmlgen/.
