Per-Source Article Count & TUI Audit Design


Goal


Give every source its own article_count setting stored in its YAML config. Ask users for a count when adding a new RSS source. Make the count editable via TUI or manual YAML edit. Remove the "Generate Custom Source Config" option from the TUI menu (demote to developer docs). Audit and add tests for all TUI menu flows that have had no coverage since the pipx migration.


Architecture


Count is a first-class field on SourceConfig. Resolution is: explicit CLI --count N → source YAML article_count → fallback 30. The CLI --count flag sentinel changes from "30" to None to enable per-source resolution.


Bundles' default_count field is removed across all bundle-related files — per-source article_count is the single source of truth.



Section 1: YAML Schema


Source YAML changes


Every builtin config-driven YAML in capcat/sources/builtin/config_driven/configs/ gets article_count: 30 added with comments. All other existing fields get inline explanatory comments. This makes files self-documenting for users who edit them manually.


Example (bbc.yaml):

# Display name shown in the TUI and output folders.
display_name: "BBC News"

# The base URL of the source website.
base_url: "https://www.bbc.com/news"

# How many articles to fetch from this source per run. Change if needed.
article_count: 30

# Request timeout in seconds.
timeout: 10

# Minimum delay between requests to this source (rate limiting).
rate_limit: 1.0

# Whether this source supports fetching article comments.
supports_comments: false

Custom source configs


All custom source configs in capcat/sources/builtin/custom//config.yaml also get article_count: 30 and field comments added.


User-added sources


When a user adds a source via the TUI, their YAML is written with article_count set to their chosen value (or 30 if blank), always with the # Change the article count if needed comment.


bundles.yml


The default_count field is removed from every bundle entry. All remaining fields get explanatory comments. The corresponding default_count field and all references are removed from:

  • capcat/sources/builtin/bundles.yml
  • capcat/core/config/source_base.py (BundleConfig.default_count field and serialisation)
  • capcat/core/config/source_registry.py (hardcoded per-bundle default_count values)
  • capcat/core/source_system/bundle_models.py
  • capcat/core/source_system/bundle_manager.py
  • capcat/core/source_system/bundle_service.py
  • capcat/core/source_system/bundle_ui.py
  • capcat/core/source_system/bundle_validator.py


Section 2: SourceConfig Dataclass


capcat/core/source_system/base_source.py — SourceConfig gains: 
article_count: int = 30  # articles to fetch per run; overridden by CLI --count

capcat/core/source_system/source_registry.py — when loading a YAML, promote article_count from custom_config to the top-level SourceConfig field: 
article_count=int(config.get("article_count", 30)),


Section 3: Count Resolution


CLI change — capcat/cli.py


_pop_value for --count currently defaults to the string "30": 
count_str, args = _pop_value(args, "-c", "--count", default="30")

Change to use None as sentinel:


count_str, args = _pop_value(args, "-c", "--count", default=None)

Downstream code that converts count_str to int must handle None — when None, per-source resolution applies rather than a global value. This applies to both fetch and bundle command handlers in cli.py. Both currently call int(count_str) unconditionally; after the change they must guard: if count_str is not None: count = int(count_str) before passing count downstream, otherwise int(None) raises TypeError.


Fetch command — capcat/commands/fetch.py


A per-source resolution helper:


def _resolve_count(cli_count_str, source_config):
    if cli_count_str is not None:
        return int(cli_count_str)
    return source_config.article_count  # already defaults to 30

Called per-source in the fetch loop, passing each source's SourceConfig.


capcat/core/unified_source_processor.py (note: lives in capcat/core/, not capcat/core/source_system/)


process_source_articles(source_name, count, ...) already accepts count — it now receives the resolved per-source value rather than a fixed global.

Section 4: Add-Source TUI Changes


Menu item removed


"Generate Custom Source Config" is removed from the TUI menu in capcat/core/interactive.py. The capcat/scripts/generate_source_config.py script stays as a developer tool, referenced from docs and the capcat.org website for power users who fork the app.


Count question added


The new method is added in three places:

  • UserInterface Protocol in capcat/core/source_system/add_source_command.py (the contract)
  • QuestionaryUserInterface in capcat/core/source_system/questionary_ui.py (the real TUI implementation)
  • MockUserInterface in capcat/core/source_system/questionary_ui.py (the test stub)

def get_article_count(self) -> int:
    """Ask the user how many articles to fetch per run. Default: 30."""

The call is inserted in AddSourceCommand.execute() in capcat/core/source_system/add_source_command.py, after select_category() and before confirm_bundle_addition().


TUI prompt:

How many articles should this source fetch per run? [30]:

  • Blank input → 30
  • Valid positive integer → use it
  • Invalid input (zero, negative, non-integer) → reprompt: Please enter a positive number.
  • Written to YAML as: article_count: 15 # Change the article count if needed

MockUserInterface.get_article_count() returns 30 by default.

Section 5: List All Sources TUI Changes


Currently prints a flat table. After this change, the list is interactive — selecting a source opens a detail view:


Source: MIT News
  Category:      science
  RSS URL:       https://news.mit.edu/rss/feed
  Article count: 30
  Bundle:        science

[E] Edit article count    [Enter] Back

Edit flow:
  1. User presses E
  2. Prompted: New article count for MIT News [30]:
  3. Validated as positive integer; reprompts on invalid input
  4. If the source is a builtin not yet mirrored to userspace, mirror it first via SourceConfigMirror (in capcat/core/source_config_mirror.py): load the manifest with _load_manifest(), call _mirror_config_driven(manifest), save with _save_manifest(manifest)
  5. Write article_count back to the source YAML in Config/sources/active/, preserving the # Change the article count if needed comment


Section 6: TUI Audit and Test Coverage


The following four menu flows have had no automated test coverage since the pipx migration. Each gets audited for breakage and covered with unit tests.


Remove Existing Sources

  • Verify EnhancedRemoveCommand path works end-to-end
  • Test: remove prompt displays available sources
  • Test: confirming removal deletes the source from userspace
  • Test: cancelling removal leaves sources unchanged

List All Sources

  • Verify sources load from userspace and display correctly
  • Test: list shows all registered sources
  • Test: selecting a source shows detail view with article_count
  • Test: edit count writes correct value to YAML

Test a Source

  • Verify it calls discover_articles() and reports results without crashing
  • Test: valid source name triggers discovery
  • Test: unknown source name shows error message, does not crash

Manage Bundles

  • Verify bundle listing works
  • Test: bundles load and display correctly
  • Test: adding a source to a bundle updates the bundle config
  • Test: removing a source from a bundle updates the bundle config


Section 7: Error Handling


  • Count input: reject values ≤ 0, non-integers. Reprompt with: Please enter a positive number.
  • YAML field missing or corrupt at read time: silently fall back to 30, log a debug warning. This covers both the single-fallback case (article_count absent, --count not provided) and the double-fallback case (--count is None and article_count missing from YAML) — both resolve to 30 via the int = 30 default on SourceConfig.
  • Builtin source not yet mirrored when editing count: call SourceConfigMirror._mirror_config_driven() first, then write
