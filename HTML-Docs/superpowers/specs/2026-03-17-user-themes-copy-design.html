Design Spec: User Themes Copy on Init + Upgrade Prompt


Date: 2026-03-17 Status: Approved

Problem


capcat init creates Config/themes/ but leaves it empty. The HTML generator always reads CSS from the package's capcat/themes/ directory and never checks Config/themes/. Users have no way to customise the visual output.

Intended UX


  1. capcat init → copies base.css + design-system.css from package into Config/themes/, writes version marker
  2. All subsequent runs → generator reads from Config/themes/ first, falls back to package if absent or on any error
  3. After pipx upgrade capcat / pipx reinstall capcat / pip install --upgrade capcat → on next run, capcat detects version mismatch and prompts:

> "Capcat themes updated in vX.Y.Z. Overwrite your Config/themes/? [Y/n]"

  • Y (default) → recopy + update version marker
  • N → update version marker only (user keeps their CSS, no re-prompt until next upgrade)

Scope


Copied to Config/themes/ on init:

  • base.css
  • design-system.css
  • Space-Grotesk/ directory (all .woff / .woff2 files) — user can swap font files in place or rewrite @font-face declarations in design-system.css to point to their own fonts

Not copied (package-owned, not user-overridable):

  • js/capcat.js — referenced via absolute package path in the generator
  • DS-logic.md — developer reference only

The upgrade prompt and version marker apply to the entire Config/themes/ contents. Answering Y recopies all three (CSS files + font directory).


Components


1. capcat/commands/init.py


  • After config_dir / "themes" mkdir, copy base.css, design-system.css, and Space-Grotesk/ from Path(__file__).parent.parent / "themes" into Config/themes/
  • Write Config/themes/.capcat-version containing the value from from capcat import __version__
  • --reinit path: never touch Config/themes/ — user-owned. check_theme_upgrade() is never called from within init_project(); it is CLI-level only

2. capcat/core/config/__init__.py — new helper


def check_theme_upgrade(project_root: Path) -> None

  • Import version as from capcat import __version__ (not from capcat.core.config which has its own __version__ = "2.0.0")
  • Reads Config/themes/.capcat-version
  • If Config/themes/ does not exist → silent no-op (uninitialised project)
  • If .capcat-version is absent or version differs from __version__ → prompt user
  • Y → recopy base.css, design-system.css, and Space-Grotesk/ from package + write new version marker
  • N → write new version marker only (suppress until next upgrade)

3. capcat/core/html_generator.py


_get_embedded_css() and _get_embedded_assets_for_template():

Both methods already compute app_dir = Path(__file__).parent.parent.absolute() (package path). Add a second lookup at that same point:


# Try user Config/themes/ first
try:
    project_root = find_project_root()
    user_css = project_root / "Config" / "themes" / "base.css"
    base_css_path = user_css if user_css.exists() else app_dir / "themes" / "base.css"
except NoProjectError:
    base_css_path = app_dir / "themes" / "base.css"

Per-file fallback: if a file does not exist in Config/themes/, fall back to the package copy for that individual file. A missing file in an existing Config/themes/ directory is not an error — it falls back silently per file.


DesignSystemCompiler constructor:
html_generator.py instantiates DesignSystemCompiler() with no arguments at line ~39. DesignSystemCompiler.__init__ hard-codes self.themes_dir = themes_dir or Path(__file__).parent.parent / "themes". To make design-system.css user-overridable:
  • Resolve user themes dir once in HTMLGenerator.__init__() via find_project_root() (catch NoProjectError)
  • Pass the resolved dir to DesignSystemCompiler(themes_dir=user_themes_dir) if user_themes_dir / "design-system.css" exists; otherwise pass nothing (keeps package default)

4. capcat/cli.py


In the pre-dispatch block (after _auto_init() but before command dispatch):


try:
    from capcat.core.config import find_project_root, check_theme_upgrade, NoProjectError
    check_theme_upgrade(find_project_root())
except NoProjectError:
    pass  # Not in a capcat project — skip silently

Skip for commands that do not require a project: init, help, version. check_theme_upgrade() is never called from init_project() itself — only from the CLI dispatch layer.


Data Flow


capcat init └─ copy capcat/themes/{base,design-system}.css → Config/themes/ └─ write Config/themes/.capcat-version = "1.0.42" (from capcat.__version__) capcat fetch hn ├─ cli.py: find_project_root() → check_theme_upgrade() → "1.0.42" == "1.0.42" → no-op ├─ HTMLGenerator.__init__: resolves user_themes_dir = Config/themes/ ├─ DesignSystemCompiler(themes_dir=Config/themes/) → reads user design-system.css └─ _get_embedded_css(): Config/themes/base.css exists → embeds it pip install --upgrade capcat (now 1.0.50) capcat fetch hn ├─ cli.py: check_theme_upgrade() → "1.0.42" ≠ "1.0.50" │ → prompt → Y → recopy + write "1.0.50" │ or → N → write "1.0.50" (suppress next run) └─ html_generator: embeds updated (or preserved) user CSS

Error Handling


  • NoProjectError anywhere → catch, fall back to package themes silently
  • File read error on any Config/themes/ file → log warning, fall back to package copy for that file
  • Missing .capcat-version in existing Config/themes/ → treat as version mismatch, prompt
  • Interrupted capcat init (directory exists, file missing) → per-file fallback in generator; check_theme_upgrade() will re-prompt on next run (.capcat-version absent = mismatch)

Testing (tests/unit/test_user_themes.py)


  • init_project() copies base.css and design-system.css to Config/themes/
  • init_project() writes .capcat-version matching capcat.__version__
  • init_project(reinit=True) does not touch Config/themes/
  • check_theme_upgrade(): version match → no prompt, no file changes
  • check_theme_upgrade(): version mismatch + Y → files recopied, .capcat-version updated
  • check_theme_upgrade(): version mismatch + N → files unchanged, .capcat-version updated to current version (suppresses re-prompt)
  • check_theme_upgrade(): .capcat-version absent in existing Config/themes/ → treated as mismatch, prompts
  • check_theme_upgrade(): Config/themes/ absent → no-op, no exception
  • _get_embedded_css(): Config/themes/base.css exists → embeds user file
  • _get_embedded_css(): Config/themes/base.css absent, dir exists → falls back to package file
  • _get_embedded_css(): NoProjectError → falls back to package file
  • DesignSystemCompiler receives user themes_dir when Config/themes/design-system.css exists
  • DesignSystemCompiler receives package themes_dir when Config/themes/design-system.css absent

Constraints


  • Never overwrite Config/themes/ on --reinit
  • Fall back to package themes silently on any per-file error — never break generation
  • Prompt only once per version bump (N answer updates version marker to suppress re-prompt)
  • check_theme_upgrade() is CLI-level only — never called from init_project()
  • base.css, design-system.css, and Space-Grotesk/ fonts are user-overridable; JS remains package-owned
  • Version comparisons always use from capcat import __version__ (not capcat.core.config.__version__)
