Part Two — 02. The New Problem


Narrative: The tool worked. But two walls existed simultaneously.
Wall 1 — External (installation): A new user who found capcat.org could not pip install capcat. They had to clone the repository, navigate to the Application folder, and run ./capcat from that exact path. One wrong directory and nothing worked. The tool was effectively invisible to anyone who did not already know how to find it.
Wall 2 — Internal (config ceiling + separation of concerns): Existing users who wanted to tune the tool — different article counts per source, per-source media settings, their own themes — had nowhere to put that configuration without editing the application's own files. News output, single article output, and the application itself all lived in the same folder. There was no separation between the tool and the archive it created.
Diagram: 01-installation-evolution — Before/After. Left: clone repo → navigate to Application/ → ./capcat. Right: pipx install capcat → cd ~/my-vault → capcat init → capcat catch. Show the wall removed.

Part Two — 03. Discovery and User Insights


Narrative: I ran user testing with four people: two developers, one ML engineer, one technically capable designer. Same methodology as Part One — map failures before designing solutions.
Three friction walls identified:

| Friction | User type who hit it | Failure moment |

|---|---|---|

| Installation required cloning the repo | All testers | First contact — couldn't get past setup |

| No per-source config customization | Developers, ML engineer | After first successful run — wanted to tune counts and media |

| Application, News, and Capcats in one folder | All testers | After using it for a week — content and tool entangled |


Two feature requests from specific users:
  • ML engineer uses arxiv.org extensively. Needed PDF downloads as a first-class feature, not a workaround.
  • Obsidian users needed frontmatter metadata (tags, source, date) and backlinking between articles and their HN/LB comment threads for knowledge graph navigation.

Part Two — 04. The Vault


Narrative: The solution to both walls was one architectural decision: separate the tool from the archive. The user owns the vault. Capcat is a tool that fills it.
The vault is the user's directory. It holds: 
~/my-vault/
  capcat.yml              ← source list and per-source counts
  Config/
    Global-settings.yaml  ← network, processing, logging, PDF settings
    sources/
      active/
        hn/config.yaml    ← per-source overrides
        lb/config.yaml
  News/                   ← batch fetch output
  Capcats/                ← single article output

The application is installed globally via pipx into an isolated environment. It never touches the vault's Config folder except to write templates on capcat init. The user can upgrade Capcat, reinstall it, or uninstall it entirely — the vault survives unchanged.


capcat init scaffolds the vault on first run. It writes capcat.yml with the default source list and Config/Global-settings.yaml with the full settings template. Every key is commented. The user edits what they want, ignores what they don't.
Separation of concerns:
  • Before: Application/, News/, Capcats/ in the same parent folder. The tool and the archive were peers.
  • After: The tool is global. The vault is wherever the user wants it.

Heuristic:

> H3, User Control and Freedom: Users often choose system functions by mistake and need a clearly marked emergency exit to leave the unwanted state without having to go through an extended dialogue.


Applied: The vault is unconditional user ownership. No sync layer. No cloud dependency. No service that can shut down and take the archive with it. The user controls the directory. Capcat is a visitor.

Part Two — 05. Configuration as UX


Narrative: Every customization decision in Capcat resolves through four levels. The user can go as deep as they need. They never have to go deeper than they want.
The four levels, highest to lowest priority:
  1. CLI flag — --pdfs, --count 20, --media. Overrides everything. Gone after the run.
  2. TUI prompt — interactive select menu at runtime. Overrides config for that session.
  3. Per-source config — Config/sources/active/hn/config.yaml. Permanent source-level override.
  4. Global settings — Config/Global-settings.yaml. Vault-wide defaults.

A user who never edits a config file gets sensible defaults. A user who wants HN to fetch 50 articles and IEEE to fetch 5 sets that once in two config files and forgets about it. A developer running a cron job overrides everything with CLI flags and pipes the output. The same hierarchy serves all three.


capcat settings --force regenerates the Global-settings.yaml template when needed. Every key is documented inline. The user never needs to read external documentation to understand what a setting does.
Heuristic:

> H7, Flexibility and Efficiency of Use: Accelerators, unseen by the novice user, may often speed up the interaction for the expert user such that the system can cater to both inexperienced and experienced users.


Applied: The novice opens Global-settings.yaml, reads the comments, changes one number. The power user maps every source to a specific count and media policy. The developer ignores config entirely and passes flags. The hierarchy is invisible until you need it.


Diagram: 06-configuration-hierarchy — 4-level resolution flow. Top: CLI flag. Second: TUI prompt. Third: per-source config.yaml. Bottom: Global-settings.yaml. Arrow direction shows resolution priority (top overrides bottom). Side labels show the user profile for each level.

Part Two — 06. New Interactive Moments


Narrative: Three new surfaces appeared between v1.0 and v1.9. Each one is a design decision made under the same constraints as Part One: technically capable users, no time, no tolerance for surprises.

Moment 1: The PDF TUI Prompt

When running interactively with sources that have PDFs, the TUI shows a speed warning before the select menu. The user sees the options — Yes, No, Source defaults — and makes one choice. That choice sets the PDF policy for the entire run.


  ⚠ PDF downloads can significantly slow article fetching.
    Recommended: use --pdfs flag for dedicated PDF runs.

  Download PDFs for this session?
  > Yes
    No
    Source defaults

"Source defaults" hands the decision back to the per-source media: block. The user who has already configured arxiv to download PDFs and Nature to skip them gets exactly that without re-specifying it interactively.


Heuristic:

> H6, Recognition Rather Than Recall: Minimize the user's memory load by making objects, actions, and options visible. The user should not have to remember information from one part of the dialogue to another.


Applied: The three options are visible. The user does not need to remember the --pdfs flag or what "Source defaults" means — the menu surfaces all three paths at the moment of decision.


Diagrams:
  • 08-pdf-resolution-flow — 4-level hierarchy: --pdfs CLI flag → TUI select → per-source media: block → global media.download_pdfs → AsyncPDFManager
  • 09-pdf-tui-prompt-flow — User flow through the interactive moment: speed warning → select menu → async background drain → progress state change logged


Moment 2: Source Update Ownership

Sources ship with Capcat. When a new version adds or changes a source config, the application needs to update that config in the user's vault. But not all source configs belong to the application. A user who has customized their hn/config.yaml should never have their changes silently overwritten.


The ownership model resolves this. Every source in the vault manifest carries an ownership field: app or user. App-owned sources update silently. User-owned sources surface an interactive questionary.select prompt:


  Source 'hn' config has been updated in the new version.
  Your vault config differs from the new app default.

  > Keep my version
    Accept app update
    Show diff

The user who never touched the config never sees this prompt. The user who customized it sees it exactly once per update.


Heuristic:

> H9, Help Users Recognize, Diagnose, and Recover from Errors: Error messages should be expressed in plain language, precisely indicate the problem, and constructively suggest a solution.


Applied: The prompt appears only when a conflict exists. It identifies the source, explains what changed, and offers three clear paths. Nothing is decided without the user's consent. Nothing is lost without the user's knowledge.


Diagram: 07-source-update-ownership — Decision tree: source detected in update → check ownership flag → app: silent update → user: interactive prompt → Keep / Accept / Diff.

Moment 3: Frontmatter and Backlinking

Every article Capcat writes includes YAML frontmatter:


---
title: The article title
source: hn
url: https://example.com/article
date: 2026-04-18
tags: [tech, programming]
comments: [[hn-comments-article-title]]
---

The comments file links back to the article:


---
title: HN Discussion — The article title
article: [[article-title]]
source: hn
---

Obsidian reads both files natively. The knowledge graph shows the article connected to its comment thread. The user who built five fragmented collections in Part One now has a single vault where every piece of content is connected by the tool that fetched it.


Heuristic:

> H2, Match Between System and the World: The system should speak the users' language, with words, phrases, and concepts familiar to the user, rather than system-oriented terms.


Applied: Obsidian users already think in linked notes and graph views. Capcat writes the wikilinks and frontmatter that Obsidian already understands. The tool did not invent a linking system. It learned the language the user already spoke.


Comment anonymization is a constraint enforced at write time. Author names in HN and LB comment threads are replaced with "Anonymous" in the saved Markdown. Profile links are preserved for reference. Privacy is built into the output, not added as an option.
Diagram: 10-frontmatter-backlinking — Article node → frontmatter block (tags, source, date, comments link) → comment file node → wikilink backlink → Obsidian graph view representation.

Part Two — 07. The Integrated Solution


Narrative: The full user journey from empty machine to running vault in four steps. 
pipx install capcat
cd ~/my-vault
capcat init
capcat catch

Step 1 installs the tool globally into an isolated environment. No cloning. No path management. No Python version conflicts.


Step 2 is the user choosing where their vault lives. It can be their Obsidian vault. It can be a Synology drive. It can be any directory. The tool goes wherever the content needs to be.


Step 3 scaffolds everything: capcat.yml with the default source list, Config/Global-settings.yaml with the full commented template, the directory structure for News and Capcats output. One command. The vault is ready.


Step 4 launches the TUI. The six-option main menu appears. The user picks a bundle, decides on HTML, reviews, confirms, and executes. The first batch of articles arrives in News/news_18-04-2026/.


From that point, the user who wants to tune per-source counts edits Config/sources/active/hn/config.yaml. The user who wants PDFs from arxiv adds media: download_pdfs: true to the arxiv source config or passes --pdfs at runtime. The user who wants to automate everything writes a cron job: cd ~/my-vault && capcat bundle tech --count 20 --html. Nothing requires touching the application files. The vault handles the rest.


Diagram: 11-user-journey-comparison-v2 — Before path (v1.0): clone repo → navigate to Application/ → ./capcat catch → manual config edit in app folder → News and Capcats in same parent as Application. After path (v1.9): pipx install → cd vault → capcat init → capcat catch → Config/ in vault → News/ and Capcats/ in vault. Key insight label: "Tool is global. Archive is yours."

Part Two — 08. Validation and Impact


Format: Same as Part One section 07. Six heuristics. Each one quoted in full, followed by one paragraph of implementation scope.
Heuristics to cover:
H4 — Consistency and Standards

Quote + scope: pipx install capcat follows the installation pattern of every modern Python CLI tool. A developer who has installed black, ruff, or httpie knows exactly what to expect. The learning curve is zero. The standard is the design.


H3 — User Control and Freedom

Quote + scope: The vault is unconditional. The tool is a visitor. The user can upgrade Capcat, replace it, or uninstall it. The archive is a directory. It survives anything.


H7 — Flexibility and Efficiency of Use

Quote + scope: Four config levels. The novice uses one. The power user uses all four. The developer uses none and passes flags. The hierarchy is invisible until needed, present when required.


H9 — Help Users Recognize, Diagnose, and Recover

Quote + scope: Source updates that conflict with user customizations surface an explicit prompt with three choices. Updates that do not conflict are silent. The user who customized nothing is never interrupted. The user who customized everything is never surprised.


H6 — Recognition Rather Than Recall

Quote + scope: The PDF prompt surfaces three options at the moment of decision. The user does not need to remember the --pdfs flag. The async drain logs state changes — the user can see when downloads start, when they complete, without watching a progress bar.


H2 — Match Between System and the World

Quote + scope: Frontmatter and wikilinks are Obsidian's native language. Capcat writes in that language. The knowledge graph connection between an article and its HN comment thread is not a Capcat feature — it is Obsidian doing what it already does, with content Capcat structured for it.
