Userspace Source Config Mirror — Design Spec


Date: 2026-03-20 Status: Approved

Problem


Builtin source configs live inside the installed package. Users cannot edit them without modifying the package installation, and edits are silently lost on upgrade. capcat add-source also writes to the installed package dir, making user-added sources upgrade-unsafe. There is no accessible, stable place for users to manage their sources, custom source configs, or bundle definitions.



Goal


Make Config/sources/active/ the single live source of truth for all source management after a project's first fetch. The installed package becomes read-only initial truth only. All CLI/TUI source commands (add-source, remove-source, list, enable/disable, bundle management) operate on userspace exclusively.


Config/themes/ is unaffected — this feature touches only Config/sources/.

Core Principle


Installed package = initial truth. Config/sources/active/ = live truth.

Terminology


  • source_id: stem of a config_driven filename. For normal files: Path(filename).stem (e.g. bbc.yaml → bbc). For disabled files: strip .disabled first — Path(Path(filename).stem).stem (e.g. skysports.yaml.disabled → skysports).
  • active builtin files: files in the builtin location that are not .disabled. For config_driven: extensions .yaml, .yml, .json. For custom: any file in the source subdirectory. For bundles: all .yml files at capcat/sources/builtin/ root.
  • manifest key: a relative path string used as the hash manifest key. Format: "<domain>/<relative-path>". Examples: "config_driven/configs/bbc.yaml", "custom/hn/config.yaml", "custom/hn/source.py", "bundles/bundles.yml". This flat-key approach handles multi-file entries (custom sources) consistently.
  • hash computation: SHA-256 of raw file bytes. Consistent across all domains.
  • tui_mode: bool, read from capcat.core.tui_context.is_tui_active() at the call site. Controls prompt style throughout.


Flows


1. First-run mirror (triggered on first capcat fetch)


Condition: Config/sources/active/ does not contain any of the three domain subdirectories.
Steps:
  1. Create Config/sources/active/config_driven/configs/, Config/sources/active/custom/, Config/sources/active/bundles/
  2. Copy all three domains:
    • config_driven: copy all .yaml, .yml, .json files (no .disabled) from capcat/sources/builtin/config_driven/configs/ → Config/sources/active/config_driven/configs/
  • custom: for each subdirectory in capcat/sources/builtin/custom/ (e.g. hn/, lb/): copy the entire directory to Config/sources/active/custom/<name>/ including config.yaml and source.py
  • bundles: copy all .yml files from capcat/sources/builtin/ root → Config/sources/active/bundles/
  1. Compute SHA-256 hash (raw bytes) of every copied file; write initial manifest to .capcat/source_hashes.json with both builtin_hash and user_hash set to that SHA-256 hash (files are identical at this point)
  2. Print/display: "Capcat: source configs written to Config/sources/active/ — edit them to customise your sources."
  3. Registry discovers custom sources from Config/sources/active/custom/ (no registry change needed — _discover_custom_sources() already scans _user_path / "custom"). Bundle service reads from Config/sources/active/bundles/bundles.yml (see Components section).

Error handling: If any domain copy fails, log a warning and continue fetch using builtins for that domain. Never block a fetch.

2. Upgrade diff (every capcat fetch after mirror exists)


Condition: Config/sources/active/ exists with at least one domain subdirectory.
Edge case — manifest missing: If any domain dir exists but .capcat/source_hashes.json is absent: re-sync. For each file currently in any user mirror location, compute its SHA-256 and set both builtin_hash and user_hash to that value. For user-added files with no builtin counterpart, set builtin_hash: "". Write the rebuilt manifest. Log: "Capcat: source_hashes.json missing — rebuilt from current state." Do not overwrite any user files.
Step 1 — Detect new items (all domains)

For each domain, collect manifest keys present in builtins but absent from the user mirror:

  • config_driven: source_ids in builtin/config_driven/configs/ but absent from Config/sources/active/config_driven/configs/
  • custom: subdirectory names in builtin/custom/ but absent from Config/sources/active/custom/
  • bundles: .yml filenames at builtin/ root but absent from Config/sources/active/bundles/

If any new items found, prompt once (CLI: print + input; TUI: questionary.confirm):

Capcat: X new item(s) available:
  Sources: guardian, ieee
  Custom sources: (none)
  Bundles: science_sources.yml
Add to your configuration? [Y/n]
  • Yes: copy new items to user mirror, add manifest entries
  • No: do nothing, no manifest update; prompt recurs on every fetch until items are added

Step 2 — Detect changed builtins (all domains)

Iterate over every manifest key with builtin_hash != "" that has a builtin counterpart:


For each manifest key:

  • Compute current_builtin_hash = SHA-256 (raw bytes) of the builtin file
  • Locate the user file:
  • For config_driven keys: check <source_id>.yaml, <source_id>.yaml.disabled, <source_id>.yml, <source_id>.json in order; use first found
  • For custom and bundles keys: the path is explicit (e.g. custom/hn/source.py) — check directly
  • If user file not found (manually deleted): remove manifest entry, log debug notice, skip
  • Compute current_user_hash = SHA-256 (raw bytes) of the found user file
  • Load stored_builtin_hash and stored_user_hash from manifest

Note on disabled config_driven sources: Enable/disable renames <source_id>.yaml ↔ <source_id>.yaml.disabled. The ordered lookup above handles this. Manifest is not modified by enable/disable.
Note on builtin removal: If a manifest key has no current builtin counterpart, do nothing to user file. Log debug notice.
Step 3 — Offer override for unmodified changed items

Collect all manifest keys satisfying the third condition. Group by domain for clarity in the prompt:

Capcat: X item(s) have updates available:
  Sources: bbc, mashable
  Custom sources: hn/config.yaml
  Bundles: bundles.yml
Override with new defaults? Your files are unmodified.
Backup will be created at Config/sources/backup_YYYY-MM-DD/. [Y/n]

(CLI: print + input; TUI: questionary.confirm)


  • Yes:
  1. Create Config/sources/backup_YYYY-MM-DD/. If exists, use backup_YYYY-MM-DD-2, -3, etc.
  2. Copy each affected user file into the backup dir (reuse the resolved path from Step 2 — do not reconstruct as <source_id>.yaml)
  3. Write new builtin content to the resolved user file path (preserves disabled state and original extension)
  4. Update manifest: set builtin_hash and user_hash to current_builtin_hash for each overridden key
    • No: do nothing to user files; update stored builtin_hash to current_builtin_hash for each declined key (prevents re-prompting for the same builtin version)

Error handling: If backup creation fails, abort override entirely. Warn user. Never overwrite without a successful backup.

add-source before first mirror: If run before capcat fetch, add-source creates Config/sources/active/config_driven/configs/ and writes the file. On next fetch, the first-run mirror condition ("directory absent") is false — upgrade diff runs instead and prompts for all builtins as new sources. This is intentional.

remove-source manifest cleanup: Remove the manifest key after successful removal. If a mirrored builtin source is removed, it reappears in the new-source prompt on next fetch. Intentional.

Hash Manifest


Location: .capcat/source_hashes.json


Keys are relative paths from the builtin root, prefixed by domain. This flat format handles multi-file entries (custom sources) without nesting.


{
  "config_driven/configs/bbc.yaml": {
    "builtin_hash": "sha256-of-builtin-bbc.yaml",
    "user_hash": "sha256-of-user-bbc.yaml"
  },
  "config_driven/configs/mashable.yml": {
    "builtin_hash": "sha256-of-builtin-mashable.yml",
    "user_hash": "sha256-of-user-mashable.yml"
  },
  "custom/hn/config.yaml": {
    "builtin_hash": "sha256-of-builtin-hn-config.yaml",
    "user_hash": "sha256-of-user-hn-config.yaml"
  },
  "custom/hn/source.py": {
    "builtin_hash": "sha256-of-builtin-hn-source.py",
    "user_hash": "sha256-of-user-hn-source.py"
  },
  "bundles/bundles.yml": {
    "builtin_hash": "sha256-of-builtin-bundles.yml",
    "user_hash": "sha256-of-user-bundles.yml"
  },
  "config_driven/configs/my-custom-source.yaml": {
    "builtin_hash": "",
    "user_hash": "sha256-of-user-added-source"
  }
}

  • builtin_hash: "" = user-added, no builtin counterpart
  • current_user_hash == stored_user_hash → unmodified → safe to offer override
  • current_user_hash != stored_user_hash → user-edited → never touch

Updated on: first-run mirror, manifest re-sync, new item addition, accepted override, declined override (builtin_hash only), removal (entry deleted).

Components


New: capcat/core/source_config_mirror.py — SourceConfigMirror


Constructor: SourceConfigMirror(project_root: Path, tui_mode: bool)
Public methods:
  • is_mirrored() -> bool — any domain dir exists under Config/sources/active/
  • run_first_mirror() — copy all three domains, write manifest, print message
  • check_for_upgrades() — diff all domains, prompt, apply

Private methods:
  • _compute_hash(path: Path) -> str — SHA-256 of raw bytes
  • _load_manifest() -> dict — read .capcat/source_hashes.json; return {} if absent
  • _save_manifest(manifest: dict) — write .capcat/source_hashes.json; creates .capcat/ with mkdir(parents=True, exist_ok=True) before writing
  • _backup(keys: list[str]) -> Path — copy affected user files to timestamped dir with counter suffix on collision; raise OSError on failure. Backup filename derivation: replace every / in the manifest key with - (e.g. config_driven/configs/bbc.yaml → config_driven-configs-bbc.yaml; custom/hn/config.yaml → custom-hn-config.yaml; bundles/bundles.yml → bundles-bundles.yml). This rule is collision-free across all three domains. The caller (check_for_upgrades) catches the OSError, prints a user-facing warning, and aborts the override — never touching any user file.
  • _builtin_config_driven_dir() -> Path — Path(__file__).parent.parent / "sources" / "builtin" / "config_driven" / "configs"
  • _user_config_driven_dir() -> Path — project_root / "Config" / "sources" / "active" / "config_driven" / "configs"
  • _builtin_custom_dir() -> Path — Path(__file__).parent.parent / "sources" / "builtin" / "custom"
  • _user_custom_dir() -> Path — project_root / "Config" / "sources" / "active" / "custom"
  • _builtin_bundles_dir() -> Path — Path(__file__).parent.parent / "sources" / "builtin" (root .yml files)
  • _user_bundles_dir() -> Path — project_root / "Config" / "sources" / "active" / "bundles"

Call site: unified_source_processor.py — at start of _process_with_new_system(), before article discovery. Pass tui_mode=is_tui_active().

Modified: capcat/core/source_system/source_registry.py — minimal change required


SourceRegistry.__init__ already sets _user_path = project_root / "Config" / "sources" / "active" when project_root is provided (existing code, no change needed there).
Change required: get_source_registry() currently constructs SourceRegistry() with no arguments and caches it in _global_registry. Update it to accept project_root: Path = None. Singleton invalidation strategy: when project_root is None, return the cached singleton as before (backward-compatible). When project_root is non-None, always construct and return a fresh SourceRegistry(project_root=project_root) — do not cache or use the cached instance. This avoids stale-singleton bugs without requiring explicit cache resets at call sites.
_discover_custom_sources() registers modules under a fixed name f"sources.custom.{source_name}.source". Since discover_sources() clears _sources and _configs at the start, and the loader explicitly overwrites sys.modules[module_name] before executing the file, the user-pass registration correctly overwrites the builtin-pass registration. User custom source files take effect.

Modified: capcat/core/source_system/source_factory.py — minimal change required


get_source_factory() currently constructs SourceFactory() with no registry and caches it in _global_factory. Update to accept project_root: Path = None. Apply the same singleton strategy: when project_root is None, return the cached singleton; when non-None, always construct and return a fresh SourceFactory(registry=get_source_registry(project_root=project_root)).

Modified: capcat/core/unified_source_processor.py — minimal change required


UnifiedSourceProcessor.__init__ calls get_source_factory() with no arguments. Update to accept project_root: Path = None and pass it to get_source_factory(project_root=project_root).
Call site chain: SourceConfigMirror is called from _process_with_new_system() with the project root already known at that point. That same project_root must be passed to UnifiedSourceProcessor at construction time, which propagates it to the factory and registry.

The cli.py list command constructs SourceRegistry() directly (no project root) — update to SourceRegistry(project_root=Path.cwd()) or pass the resolved project root.

get_available_bundles() gains an optional project_root: Path = None parameter. When provided and project_root / "Config" / "sources" / "active" / "bundles" / "bundles.yml" exists, read from there; otherwise fall back to the installed package builtin. Signature: def get_available_bundles(project_root: Path = None) -> dict.

Call sites that must be updated to pass project_root:

  • update_manager.py — UpdateManager receives project root at construction; pass it through to the call
  • interactive.py — pass project root (already available as a module-level constant or via the same mechanism used by SourceConfigMirror)

BundleService is already constructed with an explicit path in interactive.py:719 — currently hardcoded incorrectly to capcat/sources/active/bundles.yml (package-relative, not project-relative). Update to project_root / "Config" / "sources" / "active" / "bundles" / "bundles.yml", falling back to the builtin path if the user file does not exist. BundleManager already accepts a path string — no structural change to BundleService needed.

Modified: add_source_service.py, remove_source_command.py / enhanced_remove_command.py

  • remove_source_command.py / enhanced_remove_command.py: redirect delete target; remove manifest entry. Existing SourceBackupManager / QuestionaryRemovalUI retained unchanged.


Out of Scope


  • Merging or diffing YAML/Python content at field/line level — file-level hash only
  • Per-item granular prompts — batched by domain
  • Syncing user-added sources back to builtins
  • Mirroring specialized/ sources (twitter, youtube, medium, vimeo, substack) — these are utility sources, not user-configurable
  • Any changes to Config/themes/


Success Criteria


  1. Fresh project: first capcat fetch creates all three domain dirs under Config/sources/active/ and copies all active builtins; .disabled files not copied; manifest written
  2. capcat add-source writes to Config/sources/active/config_driven/configs/; manifest entry has builtin_hash: ""
  3. Bundle commands read/write Config/sources/active/bundles/bundles.yml; fall back to installed package if user file absent
  4. Upgrade diff correctly handles new items, changed-unmodified (offer override with backup), changed-modified (skip), absent user file (remove manifest entry)
  5. Override backup created before any write; aborted if backup fails; same-day collision uses counter suffix
  6. New unit tests cover:
    • First-run mirror: all three domains copied, no .disabled, manifest written
  • config_driven upgrade: new source prompt, changed+unmodified override, changed+modified skip
  • custom upgrade: config.yaml changed+unmodified → offered; source.py changed+unmodified → offered independently
  • bundles upgrade: bundles.yml changed+unmodified → offered
  • Manifest missing: rebuilt, no overwrites
  • User file manually deleted: manifest entry removed, not re-copied
  • Backup collision: counter suffix
  • add-source before first mirror: dir created, file written, builtin_hash: ""
  • remove-source: file deleted, manifest entry removed, source reappears in new-source prompt
  • Disabled source (.yaml.disabled) participates in upgrade diff hash comparison
  • Builtin removed from package: user file untouched, debug log only
  • TUI mode: prompts use questionary.confirm; CLI mode: use print + input
