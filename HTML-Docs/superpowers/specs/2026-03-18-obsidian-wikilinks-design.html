Obsidian Wikilinks: Article ↔ Comments Design


Date: 2026-03-18 Status: Approved

Goal


Connect Article-Title.md and Article-Title-Comments.md bidirectionally in Obsidian's graph view. Both files get wikilinks at top and bottom, matching the HTML navigation pattern already present in the HTML output.


Approach: B2 — Post-write injection


Article is written first (no wikilink). After fetch_comments returns True, the article file is opened and the comments wikilink is injected at top and bottom. The comments file gets the article wikilink baked in at generation time (safe — article always exists when comments are written).


Rejected approach: writing the article→comments link at article write time would create dead wikilinks for articles where comments fetch fails (network error, 403). B2 produces a clean graph with no unresolved nodes.


Wikilink Format


← [[My-Article-Title|Article]]              (top of comments.md)
← [[My-Article-Title|Article]]              (bottom of comments.md)

→ [[My-Article-Title-Comments|Comments]]    (top of article.md)
→ [[My-Article-Title-Comments|Comments]]    (bottom of article.md)

Plain text arrows (← →). Labeled wikilinks so Obsidian displays "Article" / "Comments" in reading view. No emojis.


Wikilink stems must match the exact filename on disk:

  • Article stem: derived from find_article_md(folder).stem
  • Comments stem: derived from find_comments_md(folder).stem

Note: find_article_md uses a case-sensitive -Comments suffix check. On macOS HFS+ (case-insensitive) this is safe as long as all comments files are written by comments_md_filename() (which always appends -Comments.md). This is a pre-existing constraint.


Data Flow


fetch_article_content()
  → writes article.md (no wikilink yet)

fetch_comments()
  → streamlined_comment_processor generates markdown WITH [[Article]] at top + bottom
  → writes comments.md
  → returns True

inject_comments_wikilink(article_folder_path, comments_stem)
  → find_article_md(article_folder_path) → get article file path
  → read line 1 of article.md
  → if line 1 starts with "→ [[": return True early (idempotency guard)
  → read article.md, strip trailing whitespace
  → prepend nav line + blank line
  → append blank line + nav line
  → write article.md

Components


1. capcat/core/storage_manager.py — new module-level function


Add as a module-level function (alongside find_article_md and find_comments_md), not as a method on StorageManager:


def inject_comments_wikilink(article_folder_path: str, comments_stem: str) -> bool:

Logic in order:

  1. article_path = find_article_md(Path(article_folder_path)) — if None: log warning, return False
  2. content = article_path.read_text(encoding="utf-8") — if raises: log warning, return False
  3. lines = content.splitlines() — if lines is empty or lines[0].startswith("→ [["): log debug, return True (idempotency guard)
  4. content = content.rstrip()
  5. new_content = f"→ [[{comments_stem}|Comments]]\n\n{content}\n\n→ [[{comments_stem}|Comments]]\n"
  6. Write new_content to article_path — if raises: log warning, return False
  7. Return True

Idempotency guard: Checks line 1 only. If a crash left only the bottom wikilink (partial write), the guard will not trigger — the next run will prepend the top link. The article then has one top wikilink and two bottom wikilinks. This is accepted: a duplicate bottom link in Obsidian reading view is cosmetic, not a graph error. No corrective stripping is done (complexity not worth it for an unlikely crash scenario).
Fallback note for generate_inline_comments_markdown: When article_folder_path is None, the article stem is derived as article_md_filename(article_title)[:-3]. This is best-effort — if the on-disk filename diverges from the derived stem (sanitization changes, long-title truncation), the wikilink will not resolve in Obsidian. This is acceptable for the fallback path (which only triggers if called without a folder path, not the normal call path).

2. capcat/core/streamlined_comment_processor.py — add article_folder_path parameter


New imports (add at top of file after existing imports): 
from pathlib import Path
from capcat.core.storage_manager import article_md_filename, find_article_md

Scope: HN and Lobsters markdown methods + generate_inline_comments_markdown. process_lesswrong_comments_optimized is excluded (config-driven generic source, no new-system fetch_comments override). LessWrong's call to generate_inline_comments_markdown will pass article_folder_path=None implicitly and use the fallback stem branch — no change needed there.

Three functions receive a new optional parameter:


def process_hacker_news_comments_optimized(
    self, soup, article_title: str, comment_url: str,
    article_folder_path: str = None
) -> str:

def process_lobsters_comments_optimized(
    self, soup, article_title: str, comment_url: str,
    article_folder_path: str = None
) -> str:

def generate_inline_comments_markdown(
    self, comments, article_title: str, comment_url: str,
    article_folder_path: str = None
) -> str:

In generate_inline_comments_markdown, derive article stem and inject wikilinks:


if article_folder_path:
    article_md = find_article_md(Path(article_folder_path))
    article_stem = article_md.stem if article_md else article_md_filename(article_title)[:-3]
else:
    article_stem = article_md_filename(article_title)[:-3]

wikilink = f"← [[{article_stem}|Article]]"
# Prepend: wikilink + blank line before the heading
# Append: blank line + wikilink at end

Existing two-call sites inside process_hacker_news_comments_optimized and process_lobsters_comments_optimized pass three positional args — because article_folder_path is a keyword-with-default, they remain correct without modification. Each method is updated to forward article_folder_path on its own call to generate_inline_comments_markdown:


return self.generate_inline_comments_markdown(
    comments, article_title, comment_url, article_folder_path
)

The not comments early-return branch (existing line: if not comments: return f"# Comments for: ...") is never reached in practice because both hn/source.py and lb/source.py check comments_processed == 0 after calling the processor and skip writing the file when it is zero. Since an empty comments list produces comments_processed == 0, fetch_comments returns False and injection is not called. Therefore: the early-return branch does not need a wikilink. The wikilink injection applies only to the populated-comments path (the main body of the function, after the early return).

3. capcat/sources/builtin/custom/hn/source.py and lb/source.py


In fetch_comments, markdown branch only (else: block, not the if html_mode: block). Pass article_folder_path to the processor call:


# hn/source.py — markdown branch
content = processor.process_hacker_news_comments_optimized(
    soup, article_title, comment_url, article_folder_path
)

# lb/source.py — markdown branch
content = processor.process_lobsters_comments_optimized(
    soup, article_title, comment_url, article_folder_path
)

article_folder_path here is the same-named parameter received by fetch_comments(self, comment_url, article_title, article_folder_path, html_mode=False) — it is already in scope, no additional lookup required.

The html_mode branch (if html_mode:) calls process_hacker_news_comments_html_optimized / process_lobsters_comments_html_optimized — these are out of scope, no change.


4. capcat/core/unified_source_processor.py — new system path only


Imports: unified_source_processor.py has no existing storage_manager import. Add a new top-level import line (place it after the other capcat.core imports): 
from capcat.core.storage_manager import find_comments_md, inject_comments_wikilink

Also add from pathlib import Path at module level if not already present at the top of the file — note the file has inline Path imports inside function bodies, but no top-level one; add it at the top.


The existing try/except block is replaced in full (not supplemented). Locate by content: the block starting with try: that contains source.fetch_comments(article.comment_url, article.title, article_path) and whose except clause logs "Failed to fetch comments for...". Replace the entire block with: 
try:
    self.logger.info(f"Fetching comments for: {article.title}")
    comments_written = source.fetch_comments(
        article.comment_url, article.title, article_path
    )
    if comments_written:
        comments_md = find_comments_md(Path(article_path))
        if comments_md:
            inject_comments_wikilink(article_path, comments_md.stem)
        else:
            self.logger.warning(
                f"Comments file not found after fetch_comments returned True: {article_path}"
            )
except Exception as comment_error:
    self.logger.warning(
        f"Failed to fetch comments for '{article.title}': {comment_error}"
    )

inject_comments_wikilink is also added to the storage_manager import. If inject_comments_wikilink raises unexpectedly it is caught by the existing except and logged as a warning — acceptable.
Call signature note: The concrete HN and LB fetch_comments overrides use (comment_url, article_title, article_folder_path, html_mode=False). The base class uses (article, output_dir, progress_callback=None). The existing positional call matches the concrete overrides — this pre-existing inconsistency is not changed.
Legacy path (_process_article_generic) is out of scope. Deprecated, injection not added.
