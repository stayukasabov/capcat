Graceful Shutdown on Ctrl+C — Design


Date: 2026-03-19 Status: Approved

Problem


When the user presses Ctrl+C during a multi-source batch fetch, the process does not exit immediately. progress.py's module-level SIGINT handler calls sys.exit(1), but ThreadPoolExecutor worker threads (non-daemon) continue running. The process hangs until every in-flight download completes. The user must press Ctrl+C multiple times.



Goal


One Ctrl+C → immediate stop → cancellation summary → exit 130.


In-flight article downloads are abandoned. This is intentional.



Out of Scope


  • Graceful completion of the current in-flight article
  • Retry or resume of a cancelled batch
  • Changes to single-article (capcat fetch --url) fetch path


Python Version


pyproject.toml currently declares requires-python = ">=3.8". This design uses executor.shutdown(wait=False, cancel_futures=True) which requires Python 3.9+.
Required change: bump requires-python to ">=3.9" in pyproject.toml.

Core Design Decision: Signal Handler Does NOT Touch the Executor


Calling executor.shutdown() from inside a signal handler is unsafe — ThreadPoolExecutor.shutdown() acquires an internal lock, and if a worker thread holds that lock when the signal fires, the call deadlocks on the main thread.


Solution: The signal handler only sets the shutdown event (already what GracefulShutdown._signal_handler does after this spec's changes). The as_completed() loop on the main thread detects the event and breaks. The finally block then calls executor.shutdown(wait=False, cancel_futures=True) safely on the main thread. No executor interaction from the signal handler at all.

Existing Assets in shutdown.py


GracefulShutdown (lines 16–91) is a context manager. After this spec's changes:
  • __enter__: installs SIGINT/SIGTERM handlers, stores originals, sets module-level singleton
  • __exit__: restores original handlers, clears singleton
  • _signal_handler: sets shutdown_event only — no sys.exit, no time.sleep, no cleanup calls
  • should_shutdown() → self.shutdown_event.is_set()


Changes


1. pyproject.toml


Change requires-python = ">=3.8" to requires-python = ">=3.9".



2. capcat/core/shutdown.py — extend GracefulShutdown


a) Remove sys.exit(130), time.sleep(0.5), and the cleanup_func invocation block from _signal_handler

After these removals, _signal_handler only sets self.shutdown_event. The fetch loop handles exit.


Also remove the cleanup_func constructor parameter and its call block entirely. No caller in the fetch path uses it, and allowing it creates a path for executor methods to be called from the signal handler — the deadlock risk this design eliminates.


b) Add module-level singleton + accessor

At module scope (below imports):

_shutdown_instance: Optional["GracefulShutdown"] = None

def get_shutdown() -> Optional["GracefulShutdown"]:
    """Return the active GracefulShutdown instance, or None if not installed."""
    return _shutdown_instance

In __enter__, after installing handlers:

global _shutdown_instance
_shutdown_instance = self

In __exit__, after restoring handlers:

global _shutdown_instance
_shutdown_instance = None

The global keyword is required in both methods to assign to the module-level variable.


Note: register_cleanup is NOT added. The signal handler does not call executor methods — that is handled by the finally block on the main thread.

3. capcat/core/progress.py — remove module-level signal handlers


Remove both signal.signal(...) calls (SIGINT and SIGTERM). These override the handler installed by GracefulShutdown. The atexit.register(_restore_cursor) stays.



4. capcat/commands/fetch.py — wrap source loop in GracefulShutdown


from capcat.core.shutdown import GracefulShutdown
import sys

def process_sources(sources, ...):
    with GracefulShutdown() as shutdown:
        completed = 0
        for source in sources:
            if shutdown.should_shutdown():
                break
            try:
                process_source_articles(source, ...)
                completed += 1
            except SourceError as e:
                # existing error handling unchanged
                ...

        if shutdown.should_shutdown():
            print(f"\nCancelled — fetched {completed} of {len(sources)} sources.")
            sys.exit(130)

    # normal return path: GracefulShutdown.__exit__ restores original handlers here
    return {...}  # existing return dict unchanged

sys.exit(130) is called inside the with block (before __exit__ restores handlers), so the signal handler remains active until we actually exit.

5. capcat/core/unified_source_processor.py — manual executor lifecycle + shutdown check


The executor is currently used as a context manager inside with get_batch_progress(...) as progress:. Change it to manual lifecycle, keeping the nesting:


with get_batch_progress(...) as progress:
    executor = ThreadPoolExecutor(max_workers=max_workers)
    try:
        # submit all futures (existing code unchanged)
        futures = {executor.submit(...): article for article in filtered_articles}

        shutdown = get_shutdown()  # after executor is constructed

        for future in as_completed(futures, timeout=total_timeout):
            if shutdown and shutdown.should_shutdown():
                break
            # ... existing result handling (timeout, success, failure) ...

    finally:
        # Always runs — on normal exit and on shutdown break.
        # Calling shutdown(wait=False, cancel_futures=True) twice is idempotent (Python docs).
        executor.shutdown(wait=False, cancel_futures=True)

Key notes:

  • get_shutdown() is called after executor = ThreadPoolExecutor(...) (ordering constraint)
  • The finally block runs on both the normal path and the break (shutdown) path
  • The existing except TimeoutError block for as_completed's global timeout is unaffected — the break exits the for loop before as_completed raises TimeoutError, so that path is simply not reached on shutdown
  • Calling executor.shutdown() twice (once in finally, once if Python's atexit also runs it) is safe — the second call is a no-op

Import to add:

from capcat.core.shutdown import get_shutdown


Data Flow on Ctrl+C


User presses Ctrl+C
  → SIGINT fires on main thread
  → GracefulShutdown._signal_handler:
      sets shutdown_event
      returns (no sys.exit, no executor calls)
  → as_completed() loop: next iteration checks should_shutdown() → breaks
  → finally block: executor.shutdown(wait=False, cancel_futures=True) [main thread, safe]
      → queued futures cancelled; running threads abandoned
  → _process_with_new_system returns (partial results)
  → BatchProgress.__exit__ fires: stops animation thread, restores cursor
  → source loop in process_sources: next iteration checks should_shutdown() → breaks
  → process_sources: shutdown.should_shutdown() → True
      prints: "Cancelled — fetched X of Y sources."
      sys.exit(130)
  → GracefulShutdown.__exit__ IS called (sys.exit raises SystemExit; with-blocks catch it)
      restores original SIGINT/SIGTERM handlers — benign, process is terminating
  → atexit: _restore_cursor (no-op, already restored by BatchProgress)
