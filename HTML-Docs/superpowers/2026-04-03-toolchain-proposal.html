Principles


  • Skills (Skill tool) are cheap — they load instructions into the current session
  • Subagents (Agent tool) are expensive — separate process, separate token budget
  • Direct tools (Read, Edit, Write, Bash, Grep, Glob) are cheap
  • Rule: use skills for orchestration logic, direct tools for execution, subagents only when genuinely unavoidable


Chain A — Bug Fix


> Trigger: bug reported or regression discovered


1.  Read context-engineering files matching task type (CLAUDE.md trigger table)
2.  Read docs/ files pointed to by those context files
3.  Read source code involved
4.  Skill(tools:smart-debug)                     ← structured investigation
5.  Skill(superpowers:writing-plans)             ← TDD plan → docs/superpowers/plans/YYYY-MM-DD-<name>.md
6.  git checkout -b fix/<short-description>
7.  Skill(tools:tdd-red)                         ← write failing test, Bash verify FAIL
8.  Edit / Write (direct tools)                  ← minimal implementation
9.  Skill(tools:tdd-green)                       ← Bash verify PASS
10. Skill(tools:tdd-refactor)                    ← inline cleanup, direct tools only
11. commit-commands:commit
12. Skill(superpowers:verification-before-completion)
13. Skill(superpowers:finishing-a-development-branch) ← merge to main
14. /save-report

Chain B — New Feature


> Trigger: new capability or behavior requested


1.  Skill(superpowers:brainstorming)             ← scope, approach, trade-offs
2.  Skill(superpowers:writing-plans)             ← full TDD plan per task
3.  git checkout -b feat/<short-description>
4.  Per task in plan:
    a. Skill(tools:tdd-red)                      ← failing test, verify FAIL
    b. Edit / Write (direct tools)               ← implementation
    c. Skill(tools:tdd-green)                    ← verify PASS
    d. Skill(tools:tdd-refactor)                 ← clean up
    e. commit-commands:commit
5.  Skill(superpowers:verification-before-completion)
6.  Skill(superpowers:finishing-a-development-branch)
7.  /save-report

Chain C — Release / Version Bump


> Trigger: all tests green, feature or fix merged, ready to ship


1.  Read context-engineering/release.md
2.  Bash: pytest tests/unit/ -q             ← must be fully green
3.  Edit capcat/__init__.py                 ← bump version (patch/minor/major per CLAUDE.md)
4.  Bash: python3 -m capcat.tools.gen_docs ← regenerate API docs
5.  commit-commands:commit-push-pr         ← or commit-commands:commit + manual push
6.  Bash: git tag vX.Y.Z && git push origin main --tags
7.  /save-report

Chain D — Investigation Only (no fix)


> Trigger: "investigate X", "why does Y happen", anomaly without confirmed fix


1.  Read context-engineering/debugging.md
2.  Read docs/ files pointed to
3.  Skill(tools:error-trace) or Skill(tools:smart-debug)
4.  Direct tools: Grep → Read → Bash (targeted tests)
5.  Document findings:
    - If fix needed → hand off to Chain A from step 5 (plan)
    - If no fix needed → document root cause in a Report or issue
6.  /save-report

Chain E — Code Review


> Trigger: before merging a branch, or user explicitly requests review


1.  Skill(pr-review-toolkit:review-pr)     ← subagent, acceptable cost for review-only tasks
    OR
    Skill(workflows:full-review)           ← if inline review preferred
2.  Any fixes → inline with direct tools
3.  commit-commands:commit
