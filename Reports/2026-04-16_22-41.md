### v1.9.48 — Fix double-path `images/images/` bug (found in acceptance test)
- **Bug found:** tonsky.me article uses bare relative image srcs (`src="scissors.svg?t=..."`). After downloading to `images/scissors.svg`, the URL replacement left `images/scissors.svg` in the markdown. When a second concurrent future for the same URL arrived with `local_path = images/scissors_1.svg`, its variant `scissors.svg` (basename/doc-relative) matched as a SUBSTRING of `images/scissors.svg`, corrupting it to `images/images/scissors_1.svg`.
- **Fix 1:** Added document-relative path variant (`doc_relative` = image path relative to the article's base URL directory) before the basename variant. For `https://tonsky.me/blog/tahoe-icons/scissors.svg?t=...` with base `/blog/tahoe-icons/`, `doc_relative = scissors.svg` (same as basename in this case, but also with query string variant).
- **Fix 2:** Changed variant detection from bare `variant in markdown_content` to `f"]({variant})" in markdown_content` — anchors the check to markdown link context `](URL)`, preventing substring matches inside already-replaced local paths like `images/scissors.svg`.
- **Fix 3:** Added `break` after first successful variant match, preventing further variants from firing.
- Acceptance test result: 111 local image refs, 0 double-paths, 12 remote refs (all nav/text links — correct)
- 681 tests pass, tagged v1.9.48, pushed, docs regenerated

## Root Causes / Key Findings

**`images/images/` double-path:** Concurrent futures processing duplicate `<img>` tags (same URL, no dedup check in img_tags loop) downloaded the same file twice, getting different local filenames (`scissors.svg` and `scissors_1.svg`). The variant check `variant in markdown_content` was a bare substring check — `scissors.svg` matched inside the already-replaced `images/scissors.svg`, causing Strategy 3 (`str.replace`) in `_create_markdown_link_replacement` to corrupt it. Fix: anchor to `](variant)` context.

**Why no dedup for img_tags:** The existing `duplicate_found` check only applies to link_tags (anchors). Multiple `<img>` tags with the same src (common for inline icons repeated in text) each get separate entries in `all_links` and run concurrent downloads. The download function handles dedup via filename suffix (`_1`, `_2`), but the replacement logic didn't account for the second future having a different local_path for the same URL.
