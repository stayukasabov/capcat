### Bug fix: article pages not picking up user CSS ‚Äî v1.0.44

**Root cause:** `TemplateRenderer` has its own `_get_embedded_assets()` method and its own `DesignSystemCompiler` instance, completely separate from `HTMLGenerator`. Articles go through `TemplateRenderer.render_template()`, not through `HTMLGenerator._get_embedded_css()`. The original feature only fixed `HTMLGenerator` ‚Äî `TemplateRenderer` remained hardcoded to the package `base.css`.

**Fix:** Applied the same pattern to `TemplateRenderer`:
- Added `from capcat.core.config import find_project_root, NoProjectError` import
- Added `_resolve_user_themes_dir()` method
- Per-file priority check in `_get_embedded_assets()`
- `DesignSystemCompiler` receives user `themes_dir` when `Config/themes/design-system.css` exists
- 2 regression tests added to `test_user_themes.py`

Bumped to v1.0.44, PR #7, squash merge, tagged, pushed.

**Final test count:** 140 passing.

## Root Causes / Key Findings

- **CSS not applying to articles:** Two separate CSS embed code paths exist ‚Äî `HTMLGenerator` (index/listing pages) and `TemplateRenderer` (article/comments pages). The feature implementation missed `TemplateRenderer` entirely.
- **Design pattern:** Both `HTMLGenerator` and `TemplateRenderer` instantiate their own `DesignSystemCompiler` and have their own CSS read method. Any future CSS-path change must be applied to both.
