### Comment Thread Depth Feature (v1.6.4)

### Blockquote CSS Fix (v1.6.5)
- Root cause: no `blockquote` rule existed; browser default `margin: 1em 40px` applied both sides per level → at depth 6, ~280px each side, ~0px content width
- Fix: `blockquote { margin: 0.25rem 0 0.25rem 12px; padding-left: 10px; border-left: 2px solid var(--border-color); color: inherit; }`
- Applied to `Application/Config/themes/base.css` (user themes, used by template renderer) and synced to `capcat/themes/base.css` (package default)
- Verified via Playwright: `marginLeft: 12px`, content width ~540px at deepest level

### Literal `>` Characters in Comment Headers
- Root cause: old `generate_inline_comments_markdown` appended the blockquote `prefix` (`> `) as a separate `md_content +=` call on the same line as the user/profile link, producing `**Anonymous** > > ([profile])` in Markdown → rendered as literal `>` in HTML
- Code fix (797ea26): merged both `md_content +=` lines into one so prefix appears only before the username
- Bulk fix: 23 existing `comments.md` files in NewsVault had the old format; fixed with Python regex `re.sub(r'(\*\*Anonymous\*\*) ((?:> )+)\(', r'\1 (', content)`
- Released as v1.6.5

## Root Causes / Key Findings

**Double-prefix bug**: `generate_inline_comments_markdown` used two sequential `md_content +=` appends — first adding the prefixed header, then re-appending the profile link with another prefix. Fix: single combined line.
