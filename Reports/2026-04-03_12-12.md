## Accomplished

### Investigation
Traced PDF lifecycle from CLI → `ArticleFetcher` → `AsyncPDFManager`. Found three root-cause bugs.

### Fixes (v1.9.4)

**B3 — Worker thread never started**
`initialize_pdf_manager()` created `AsyncPDFManager` but never called `.start()`. The worker thread and `ThreadPoolExecutor` were never created. Downloads were queued but nothing drained the queue.
Fix: `initialize_pdf_manager` now calls `.start()` on the new instance before returning.

**B4 — Manager recreated per article**
`ArticleFetcher.__init__` called `initialize_pdf_manager(config.pdf)` on every instantiation. HN source creates a new `NewsSourceArticleFetcher` per article (~30 for a typical batch). Each call created a fresh `AsyncPDFManager`, overwriting `_global_pdf_manager` and discarding all queued downloads from prior articles.
Fix: `initialize_pdf_manager` is now idempotent — returns the existing running instance if the worker thread is alive; only creates a new instance if the manager is not running.

**B5 — No drain or shutdown after batch**
`_process_articles_with_new_system` returned as soon as article futures completed. Background PDF downloads outlive the articles loop. No wait mechanism existed.
Fix: Added `AsyncPDFManager.wait_until_idle(timeout)` method that polls until queue is empty and no active downloads. Wired `wait_until_idle(120s)` + `shutdown_pdf_manager()` into `_process_with_new_system` after `_process_articles_with_new_system` completes.
