### Bug fix: MediaProcessor.parse_srcset (v1.6.8)

**Root cause:** `MediaProcessor.parse_srcset` (lines 690–750, `capcat/core/media_processor.py`) used `srcset.split(",")` — naive comma split. Substack/Cloudinary CDN URLs embed transformation params with commas in the path (e.g. `w_1456,c_limit,f_auto,fl_progressive:steep`), producing corrupt fragments. The fragment `fl_progressive:steep/https%3A%2F%2F...` was then passed to `urljoin(article_url, fragment)`, producing `https://senykamara.substack.com/p/fl_progressive:steep/...` — a 404.

**Fix:** Replaced naive split with descriptor-aware fragment-merging algorithm (same approach as `formatter._parse_srcset` which was fixed in v1.6.7). A fragment is a continuation of the previous URL entry when it does not start a new URL scheme AND the previous entry has not yet ended with a valid descriptor (`Nw` or `Nx`).

## Root Causes / Key Findings

**`MediaProcessor.parse_srcset` naive comma split:** Three copies of the srcset parser exist in the codebase:
1. `formatter.py::_parse_srcset` — fixed in v1.6.7
2. `image_processor.py` — imports from formatter (fixed transitively)
3. `media_processor.py::MediaProcessor.parse_srcset` — independent implementation, fixed this session in v1.6.8

The `_ends_with_descriptor` helper is intentionally inlined (not imported from formatter) to keep `MediaProcessor` self-contained. The inline `try/except float()` is functionally equivalent to `formatter._is_float`.

**urljoin behavior with `fl_progressive:steep/...`:** Python's `urljoin` does NOT treat `fl_progressive:steep` as a URL scheme (unlike schemes like `http:`). It resolves it as a relative path, producing the article-base-prefixed broken URL.
